Trees are one of the most versatile and widely used data structures in computer science. A \newterm{tree} is a hierarchical data structure consisting of nodes, where each node has a value and a set of references (\newterm{edges}) to its child nodes. The node at the top of the hierarchy is called the \newterm{root}, and nodes with the same parent are called \newterm{siblings}, as seen in Figure~\ref{fig:tree1}.\\

Trees are a subset of Graphs, a data structure comprised of nodes and edges. Graphs can be converted into trees by adding the following two constraints:
\begin{itemize}
  \item The graph is connected, such that there exists a path between every pair of nodes
  \item The graph is acyclic; no cycles exist in the graph
\end{itemize}
With this comes the caveat that the number of edges in a tree is $\text{edges} = n - 1$, where $n$ is the number of nodes in the tree. This is because a tree with $n$ nodes must have exactly $n-1$ edges to maintain its connected and acyclic properties. If there were more than $n-1$ edges, it would create a cycle, violating the acyclic property. If there were fewer than $n-1$ edges, it would not be fully connected, violating the connected property.

Less formally, a \newterm{forest} is a collection of multiple trees (may be disconnected), and a \newterm{tree} is a forest with only one tree. Lots of agricultural terms in this chapter!

Trees have a unique structural property: between any two nodes, there exists exactly one simple path. This follows directly from the fact that trees are both connected and acyclic. As a result, there is only one way to move from a start node to an end node.

Although trees are often drawn with a root at the top, this is a matter of convention. Any node in a tree can be chosen as the root, and doing so induces a hierarchical structure in which edges are interpreted as parent-child relationships.

Once a root is selected, every node (except the root) has exactly one parent, and zero or more children. Because of this relationship, any randomly selected node can form a tree (we call this a \newterm{subtree}, consisting of the node and all its descendants), making the tree structure \newterm{recursive}. Moving from one node to the next is the recursive step that minimizes the size.

Nodes with no children are called, as you'd expect, \newterm{leaves}, as they are the outermost node of a tree. The \newterm{depth} of a chosen node is the number of edges from the \emph{root} to that node, and the \newterm{height} of a tree is the maximum depth among all nodes. Equivalent to the height of a tree is the maximum number of edges from the \emph{root} to any leaf. You may see the \newterm{level} of a node referred to as the number of edges from the \emph{root} to that node, similar to depth. By convention, the level and depth of the root is 0. See Figure~\ref{fig:tree2}.

Some more relevant terminology you may encounter in the context of trees:
\begin{itemize}
  \item \newterm{Path}: A sequence of nodes where each adjacent pair is connected by an edge. This can be used to show the idea of how to get between two nodes in a tree, for example, between two cities in a flight path network.
  \item \newterm{Ancestor}: A node that lies on the path from the root to a given node. This is done by repeatedly traversing from a child node to a parent node until the selected node is reached.
  \item \newterm{Descendant}: A node that lies on the path from a given node to a leaf. This is done by repeatedly traversing from a parent node to a child node until a leaf is reached.
\end{itemize}

\section{Binary Trees and Binary Search Trees}
The most common type of tree is the \newterm{binary tree}, where each node has \emph{at most} two children, referred to as the \newterm{left child} and the \newterm{right child}. Binary trees are widely used in various applications, including expression parsing, binary search trees, and heaps.

Let's put this practice. Recall our binary search implementation from the recursion chapter. Given a chosen word and a sorted list of words, such as a dictionary, we can find the page of the chosen word by repeatedly dividing the available pages in half and comparing the elements on the page to the chosen word. This is much easier than, for example, going through every single page and checking if the word is on that page.

Say we wanted to represent our dictionary as a tree that we could recursively binary search through. For simplicity, we will limit our words to \texttt{mango}, \texttt{apple}, \texttt{pear}, \texttt{banana}, \texttt{anagram}, \texttt{peony} and \texttt{peach}. These 7 words form a binary tree of height 2, as seen in Figure~\ref{fig:bst1}. We select the word \texttt{mango} as the root as it is the middle word after sorting alphabetically.

Similar to binary search, we can find any word in the tree by starting at the root and comparing the target word to the current node's word. If the target word is less than the current node's word, we move to the left child; if it is greater, we move to the right child. We repeat this process until we find the target word or reach a leaf node, which tells us that the target word is not in the tree. This structure allows us to cut the available words in half with each step in the tree we go.

\section{File Structures and File System Hierarchies}
Computers, since their first creation in the 1980s, have needed a way to store and organize data. Prior to computers, many systems of organization were used to store physical data, such as filing cabinets, libraries, and warehouses.

These systems often had a hierarchical structure, with a top-level category that branched out into subcategories, with these subcategories further branching out into more specific categories, and so on.

For example, a library might have two top-level categories: \texttt{fiction} and \texttt{non-fiction}. Within the category \texttt{non-fiction}, you may find \texttt{psychology}, \texttt{biographies}, \texttt{autobiographies}, \texttt{self-help}, \texttt{journalism} books, or even \texttt{nature guides}. Under the category \texttt{fiction}, you may find \texttt{romance}, \texttt{classics}, \texttt{crime}, \texttt{science fiction}, \texttt{thriller}, or \texttt{fantasy} novels. Of course, these sub-genres can have even further categorizations too.

A hospital may have patient records categorized by first letter of last name, and then sorted alphabetically from there.

With the development of the computer, sorting these types of hierarchical data became easy with the concept of directories. A \newterm{directory}, also called a \newterm{folder}, is a categorical way of organizing data (for example, by storing files or other subdirectories). We will stick to the more technical term of directories, however, any time you create a folder on your computer, you are making a \emph{directory}. The name comes from how, in the twentieth-century, many phone books were referred to as telephone directories.

This structure naturally fits into the tree data structure:
\begin{itemize}
  \item The \textbf{root} of the tree is the top-level directory (\texttt{/} on Linux/Unix\footnote{There is a very helpful video in Linux File Systems in your digital resources. Check it out!} systems or \texttt{C:\textbackslash} on Windows).
  \item A \textbf{node} is naturally fits the idea of either \emph{directories nodes} or \emph{files nodes}.
  \item A \textbf{directory node} may have children, where its children are subdirectories or files.
  \item A \textbf{file node} is a \textbf{leaf}, meaning it has no children.
\end{itemize}
